# 配置・展示運用・障害対応

## 推奨配置

```text
公開HTTPS環境
  Next.js Web App
  Go Backend
  SQLiteまたはPostgreSQL
          |
          | outbound WebSocket
          v
Mac mini M4 または高性能ノートPC
  Unity / TouchDesigner
          |
          | HDMI / Audio / Control
          v
プロジェクター・音響・照明

運営ノートPC
  運営ダッシュボード
  到着キュー操作

Raspberry Pi 4B
  死活監視
  NFC・物理ボタン・LED制御候補
```

学内ネットワークから描画PCへの受信ポート開放を要求せず、描画PC側から公開サーバーへ接続する。

## 機材役割

| 機材 | 主役割 | 代替 |
|---|---|---|
| Mac mini M4 | メイン描画 | GPU搭載ノートPC |
| ノートPC 1 | 運営ダッシュボード | ノートPC 2 |
| ノートPC 2 | 受付・到着操作 | ノートPC 1 |
| ノートPC 3 | バックアップ描画 | 保存済み映像再生 |
| Raspberry Pi 4B | 死活監視・外部制御 | 手動監視 |

## MVP性能目標

実地試験前の仮目標。

| 指標 | 目標 |
|---|---:|
| 同時追跡 | 50セッション |
| 位置送信間隔 | 5秒 |
| 展示反映遅延 | 2秒以内 |
| 描画WebSocket再接続 | 10秒以内 |
| 到着演出時間 | 10〜20秒 |
| 描画 | 30fps以上 |
| 連続稼働 | 8時間 |

## 起動手順

1. インターネット、学内Wi-Fi、モバイル回線を確認する
2. GoのDB接続と `/health/ready` を確認する
3. 描画機を起動し、WebSocket接続を確認する
4. プロジェクター、解像度、音量、照明を確認する
5. 運営ダッシュボードで描画機がonlineになることを確認する
6. テスト端末でQR参加、位置送信、到着、再生、停止を通す
7. フォールバック映像へ切り替えられることを確認する

## 監視項目

- 追跡中セッション数
- 1分以内に位置更新されたセッション数
- WebSocket接続状態と最終heartbeat
- 到着キュー件数と最古待ち時間
- APIエラー率
- DB容量
- 描画fps
- ネットワーク往復時間

運営ログに生の緯度経度を出さない。セッションIDも画面では短縮表示する。

## 障害対応

| 障害 | 自動処理 | 運営対応 |
|---|---|---|
| インターネット切断 | 描画は最後の状態から自律表現へ移行 | 回線確認、保存済み映像へ切替 |
| Go停止 | 描画機が再接続を継続 | サーバー再起動、ready確認 |
| 描画ソフト停止 | watchdogが検知 | バックアップ描画機を接続 |
| GPS拒否 | Web AppがQRチェックポイントを案内 | 参加を強制しない |
| GPS精度低下 | 最終有効位置を短時間保持 | QRチェックポイントを案内 |
| 到着演出滞留 | タイムアウトでskip | キューを確認し手動skip |
| プロジェクター停止 | 音響または補助表示を継続 | 電源・入力・ケーブル確認 |

## フォールバック展示

公開サーバーや位置取得が利用できなくても、作品が完全停止しない構成にする。

- 過去の匿名化済み軌跡またはテスト用生成データを再生する
- 「リアルタイム接続なし」を運営画面には表示する
- 本番スクリーン上では障害メッセージではなく自律的な抽象表現へ切り替える
- 復旧後は `display.snapshot` を受け取り、自然にリアルタイム表示へ戻る

## 終了手順

1. 新規参加受付を停止する
2. 追跡中セッションを停止する
3. 到着キューを完了またはskipする
4. 展示用クライアントを切断する
5. DBバックアップの要否を確認する
6. 保存期間に従って位置情報を削除する
7. エラー件数と運用メモだけを記録する

## リハーサル受け入れ条件

- 主要導線を3回歩き、軌跡が3D空間と対応する
- 10台以上で同時位置送信できる
- 3人同時到着を処理できる
- 通信断から表示が自動復旧する
- 追跡停止後に位置点を保存しない
- フォールバック展示へ1分以内に切り替えられる
